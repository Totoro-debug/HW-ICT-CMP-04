
# 一、检查阶段

## 任务目标
严格遵循 `/design-docs/03-通用规范与非功能设计.md`，为 `code/` 中的每一个模块分别启动一个子代理，检查各个模块的实现是否符合设计文档的要求。

## 输入
每个模块 `<module-name>` 对应的子代理 `agent-<module-name>` 的输入包含如下文件：

| 类别 | 文件路径 |
|------|----------|
| 核心设计文档 | `/design-docs/03-通用规范与非功能设计.md` |
| 模块源文件 | `/code/ecommerce-<module>/src/main/java/com/ecommerce/<module>/**/*.java` |
| 配置文件 | `/code/ecommerce-<module>/src/main/resources/*.yml`（如存在） |
| 模块 pom.xml | `/code/ecommerce-<module>/pom.xml` |
| 根 pom.xml | `/code/pom.xml` |


## 输出

每个子代理最终都必须输出一份 `/docs/03-check/<module-name>-check.md`，文件要求如下：

- **检查结论**：列出"一致"和"不一致"两大类，每类用点列形式
- **不一致项**必须包含：
  - 代码定位（如 `ecommerce-user/src/main/java/.../UserController.java:45`）
  - 设计要求定位（如 `02-系统架构.md §4 关键本地接口 - UserQueryService`）
  - 不一致的具体描述（API 签名、类名、注解、返回类型、依赖方向、Key 格式等）
  - 原因解析：为什么当前实现不符合设计
- **一致项**：简要列出已确认一致的检查点（不需要细节，但需覆盖主要维度）
- **检查遗漏声明**：如果某些维度在代码中找不到对应实现（如某事件监听器缺失），需显式说明"未找到"，不得默认略过

## 检查范围与判定标准

严格遵循 `/design-docs/03-通用规范与非功能设计.md`，仔细检查以下设计要求：

### 1. 金额计算（Section 1）
- [ ] 所有金额是否使用 `BigDecimal`，禁止使用 `double` 或 `float`
- [ ] 最终入库金额是否保留 2 位小数
- [ ] 舍入模式是否使用 `RoundingMode.HALF_UP`
- [ ] 优惠金额是否满足：不得小于 0，不得大于商品金额
- [ ] 应付金额是否满足：不得小于 0.01

### 2. 通用异常（Section 2）
- [ ] `BusinessException` — 通用业务异常，HTTP 400
- [ ] `ResourceNotFoundException` — 资源不存在，HTTP 404
- [ ] `AuthorizationException` — 未认证或无权限，HTTP 401/403
- [ ] `ValidationException` — 参数校验失败，HTTP 400
- [ ] `ConflictException` — 状态冲突、重复提交，HTTP 409
- [ ] `RateLimitException` — 触发限流，HTTP 429
- [ ] 订单金额校验失败是否抛出 `OrderValidationException`，不得抛出 `IllegalArgumentException`

### 3. 幂等规范（Section 3）
- [ ] 创建订单接口是否以 `externalOrderNo` 为幂等键
- [ ] 支付回调接口是否以 `paymentNo` + `callbackSequence` 为幂等键
- [ ] 退款申请接口是否以 `refundRequestNo` 为幂等键
- [ ] 物流回调接口是否以 `trackingNo` + `eventTime` + `status` 为幂等键
- [ ] 发票申请接口是否以 `invoiceRequestNo` 为幂等键
- [ ] 重复请求是否返回第一次处理结果，做到不重复扣款、重复扣库存、重复发积分或重复开票。

### 4. 本地限流（Section 4）
- [ ] 登录接口是否限制同一用户名每分钟 5 次
- [ ] 支付回调接口是否限制同一 paymentNo 每分钟 20 次
- [ ] 商品搜索接口是否限制同一 IP 每分钟 120 次
- [ ] 创建订单接口是否限制同一用户每分钟 20 次
- [ ] 限流触发时是否返回 429，错误码为 `RATE_LIMITED`

### 5. 黑盒测试隔离（Section 5）
- [ ] 业务代码是否**不依赖** reset/bootstrap 接口
- [ ] 是否通过正式 REST API 或 ADMIN 测试支撑接口创建前置条件

### 6. 审计日志（Section 6）
- [ ] 用户冻结和解冻是否记录审计日志
- [ ] 商品上下架是否记录审计日志
- [ ] 库存人工调整是否记录审计日志
- [ ] 订单取消审核是否记录审计日志
- [ ] 退款审核和仓库验收是否记录审计日志
- [ ] 发票开具是否记录审计日志
- [ ] 结算批次生成是否记录审计日志
- [ ] 审计日志至少包含：操作者、操作类型、业务 ID、操作前状态、操作后状态、操作时间、备注

### 7. 通知规范（Section 7）
- [ ] 所有通知是否通过 `LocalNotificationService` 发送
- [ ] 业务模块是否只提交 `NotificationRequest`，不直接调用 `MockMailSender` 或 `MockSmsSender`

### 8. 本地事件失败处理（Section 8）
- [ ] 事件监听器失败时是否记录事件处理失败日志
- [ ] 是否保存失败记录到本地事件处理表
- [ ] 是否**不回滚**主业务事务（非强一致监听器）
- [ ] 支付成功后的物流、积分、通知监听器是否均为非强一致监听器

## 要求

1. **严格遵循设计文档**：仔细检查文档中所有和当前模块相关的设计要求，不遗漏任何条款。
2. **仅检查本阶段任务**：只检查 `03-通用规范与非功能设计.md` 中的要求，其他任何 bug 或未实现的功能都不需要检查。
3. **不允许修改代码**：只能检查并给出最终输出，不得对代码做任何修改。
4. **分批次启动子代理**：每次启动 2 个子代理，防止并发量过高。
5. **及时输出结果**：每个子代理完成任务后，必须立即将结果输出到 `/docs/03-check/<module-name>-check.md`。
6. **输出完整性**：每一项不一致都需要有：设计要求定位、代码定位、详细原因解析。
7. **准确判定 Match/Mismatch**：对于每一项检查，如果设计与实现完全一致则标记为 Match，否则标记为 Mismatch 并详细说明。

## 执行顺序

建议按以下顺序分批启动子代理（每批 2 个）：

| 批次 | 模块 1 | 模块 2 |
|------|--------|--------|
| Batch 1 | ecommerce-user | ecommerce-product |
| Batch 2 | ecommerce-inventory | ecommerce-cart |
| Batch 3 | ecommerce-order | ecommerce-promotion |
| Batch 4 | ecommerce-payment | ecommerce-logistics |
| Batch 5 | ecommerce-loyalty | ecommerce-review |
| Batch 6 | ecommerce-common | ecommerce-app |




# 二、修复方案

## 任务目标

分析 `/docs/03-check` 中各模块的一致性检查报告，参考 `design-docs/03-通用规范与非功能设计.md`（以下简称"设计文档 03"），为每个不一致点制定真实可用的修复方案，最终输出汇总报告与监控清单。

## 输入材料

| 优先级 | 材料 | 说明 |
|--------|------|------|
| 1 | `/docs/03-check/*/` 下的所有一致性检查报告 | 需逐一阅读，包含各模块发现的不一致项 |
| 2 | `design-docs/03-通用规范与非功能设计.md` | 设计规范基准，包含金额计算、异常、幂等、限流、审计日志、通知、事件失败处理等要求 |
| 3 | `code/` 下的项目源文件 | 用于验证修复方案的可行性，必要时查阅具体类/方法实现 |

## 输出要求

### 输出文件 1：`/docs/03-check/resolution.md`

按点列出修复方案，每条方案需包含：

```markdown
### [不一致点编号]

**模块**：`xxx`
**问题描述**：[设计要求 vs 当前实现]
**修复方案**：[符合代码和设计的真实可用方案，需具体到类名、方法名、逻辑改动点]
**验证方式**：[可通过何种方式验证该修复有效]
```

**约束**：
- 只针对 `/docs/03-check` 中报告的不一致点给出修复方案，不扩展到其他 bug 或未实现功能
- 修复方案必须是可落地执行的真实方案，不可泛泛而谈
- 禁止修改设计文档或 API 契约

### 输出文件 2：`/docs/03-check/checklist.md`

包含所有问题的跟踪清单，格式如下：

```markdown
## 问题跟踪清单

| 序号 | 模块 | 不一致点描述 | 严重程度 | 修复方案摘要 | 状态 |
|------|------|--------------|----------|--------------|------|
| 1 | xxx | ... | 高/中/低 | ... | 待修复/已修复 |
```

**约束**：
- 包含 `/docs/03-check` 所有模块的全部不一致点
- 用于监控后续流程进度
- 状态字段初始值均为"待修复"

## 执行流程

### 第一阶段：信息收集

1. 遍历 `/docs/03-check/` 下所有子目录（每个子目录对应一个模块）
2. 阅读每个子目录中的检查报告，记录所有不一致点
3. 对照设计文档 03，理解每条规范的具体要求
4. 必要时查阅 `code/` 中相关源文件确认当前实现

### 第二阶段：方案制定

1. 针对每个不一致点，结合设计文档要求和当前代码实现，制定具体修复方案
2. 修复方案需包含：受影响的类/方法、需修改的逻辑、具体改动描述
3. 标注每个问题的严重程度（高/中/低）

### 第三阶段：结果输出

1. 按上述格式要求，将修复方案写入 `/docs/03-check/resolution.md`
2. 将问题清单写入 `/docs/03-check/checklist.md`
3. **每个子代理完成任务后，必须立即将结果追加写入输出文件**（不等待全部完成再写入）

## 子代理执行策略

1. 分批次启动子代理，每批启动 **2 个子代理**（防止并发量过高）
2. 每批子代理处理独立的模块集合，避免任务重叠
3. 各子代理完成工作后，**立即将结果写入输出文件**，后再处理下一批
4. 主流程负责汇总各子代理结果，确保无遗漏、无重复

## 约束清单

- **只读操作**：本任务仅输出修复方案，**不修改任何代码**
- **范围控制**：仅处理 `/docs/03-check` 中报告的不一致点，不扩展到其他问题
- **设计为基准**：设计文档 03 是规范基准，修复方案必须符合设计意图
- **真实可用**：修复方案需具体、可执行，不是泛泛的建议
- **及时输出**：子代理完成后立即写入结果，不批量等待




# 三、执行修复

## 任务目标

严格遵循 `/docs/03-check/resolution.md` 中的修复方案，启用子代理按模块并行修复 `code/` 中项目存在的问题，最终使代码符合设计文档 03 的规范要求。

## 输入材料

| 优先级 | 材料 | 说明 |
|--------|------|------|
| 1 | `/docs/03-check/resolution.md` | 详细的修复方案，包含每个不一致点的具体修复步骤 |
| 2 | `/docs/03-check/checklist.md` | 修复进度监控清单，需在每项修复完成后更新状态 |
| 3 | `code/` 下的项目源文件 | 实际修复操作的对象 |
| 4 | `design-docs/03-通用规范与非功能设计.md` | 设计规范基准，修复时如遇细节不清以此为准 |

## 输出要求

- **直接输出**：`code/` 下经过修复的 Java 源代码（修改应在本地文件生效）
- **间接输出**：`/docs/03-check/checklist.md` 中所有已完成项的状态更新为"已修复"

## 验收标准

1. 每个模块都必须修复该模块在 `resolution.md` 中对应的所有问题
2. 修复后的项目必须能通过测试命令：`mvn -f code/pom.xml test`
3. 修复过程不得破坏已有功能，不得修改 API 契约

## 执行流程

### 第一阶段：任务准备

1. 阅读 `/docs/03-check/resolution.md`，理解所有修复方案
2. 阅读 `/docs/03-check/checklist.md`，了解问题总量和严重程度分布
3. 按模块对修复任务分组，每批选择 2 个互不依赖的模块并行处理

### 第二阶段：并行修复

1. **启动子代理**：每批启动 **2 个子代理**，每个子代理负责一个模块的全部修复
2. **执行修复**：子代理按照 `resolution.md` 中该模块的修复方案逐项执行
3. **即时更新**：子代理每完成一项修复，立即更新 `checklist.md` 中对应条目的状态为"已修复"（不等到本模块全部完成再更新）
4. **记录阻塞**：如遇无法按方案修复的情况（代码架构不支持等），记录具体原因，标记为"待确认"

### 第三阶段：验证

1. 所有子代理完成后，运行 `mvn -f code/pom.xml test` 验证编译和测试通过
2. 如有测试失败，分析原因：
   - 若是本次修复引入的，退回相关修改，重新制定修复方案
   - 若是其他已有问题，记录到 `checklist.md` 的备注中，单独处理
3. 更新 `checklist.md` 中所有条目的最终状态

## 子代理执行策略

1. **批次控制**：每批启动 2 个子代理，当前一批全部完成后再启动下一批
2. **任务分配**：每个子代理负责一个完整模块的修复任务（如 `ecommerce-order`、`ecommerce-promotion` 等）
3. **互不重叠**：各子代理处理的模块不重叠，避免合并冲突
4. **即时回写**：子代理完成每项修复后，立即写入 `checklist.md` 更新状态，不在内存中积压
5. **主流程协调**：主流程负责任务分发、批次调度和最终验收

## 约束清单

- **范围管控**：只修复 `resolution.md` 中提及的问题及修复这些问题必然涉及的相关问题，不扩展到其他 bug 或未实现功能
- **不可逆操作谨慎**：删除文件、移动类等不可逆操作需在 `checklist.md` 中备注，并确认无测试依赖
- **设计为基准**：修复方案以设计文档 03 为准，如遇方案与设计冲突，以设计为准重新评估方案
- **API 契约不可破坏**：不得修改 REST API 的 URL、HTTP Method、Request/Response 字段名和类型、成功 HTTP 状态码
